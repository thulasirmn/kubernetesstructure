1. Executive Summary

The Secure Research Workspace (SRW) platform is built on top of Azure Trusted Research Environment (Azure TRE), which today provisions isolated Azure Storage Accounts and File Shares for each research workspace. This document describes the plan to extend the platform with the capability for researchers to launch, access, and manage containerised analytical applications — Jupyter Notebook, RStudio, and custom Docker applications — directly within their provisioned workspace, from a React-based browser UI.

Scope in one sentence: Extend the existing TRE workspace application so that every researcher assigned to a workspace can click an application icon in the browser, have a personal containerised environment spin up on AKS within minutes, and access it via a browser URL — with full data isolation, async provisioning, and clean lifecycle management.

The implementation spans five layers:

  • Terraform infrastructure — AKS namespace provisioning (new) and per-session K8s resource management
  • Backend API — ASP.NET Core Minimal API with Clean Architecture; async session launch pattern
  • Background workers — Terraform orchestration, K8s status polling, session stop, idle cleanup
  • React UI — application gallery, click-to-launch flow, session status polling, stop
  • End-to-end testing — live Jupyter, RStudio, storage isolation, stop flows

3.3 Kubernetes Resource Model

The platform manages two tiers of Kubernetes resources, each backed by a separate Terraform state file in Azure Blob storage:

TierResources CreatedLifecycleTerraform State Key
Workspace (once per workspace) kubernetes_namespace
kubernetes_secret (azure-storage-creds)
kubernetes_network_policy (default-deny)		 Created when workspace activates. Destroyed when workspace is deleted. workspaces/{k8sNamespace}-k8s.tfstate
Session (once per user per launch) kubernetes_deployment (1 replica)
kubernetes_service (ClusterIP)
kubernetes_ingress_v1 (path-based)		 Created on session launch. Destroyed on session stop or idle reap. sessions/{sessionId}.tfstate

Security: The Azure Storage Account key (from TRE) is bound into the azure-storage-creds K8s Secret at namespace creation time, passed exclusively as the environment variable TF_VAR_storage_account_key — it is never written to a .tfvars file on disk or stored in Terraform state in plaintext.

3.4 Session Lifecycle & Async Pattern

Because Terraform provisioning takes 2–3 minutes per session, the API uses an async pattern. The HTTP request returns immediately; a background worker does the slow work; the client polls for status.

Researcher clicks Launch in UI │ ▼ POST /api/workspaces/{id}/sessions │ (< 2 seconds) ▼ 202 Accepted {status: "Starting", accessUrl: null} │ ▼ (SessionLaunchWorker — background, ~2-3 min) terraform init + apply → K8s Deployment + Service + Ingress created → session.DeploymentName, ServiceName, AccessUrl written to DB │ ▼ (SessionStatusPoller — every 5 s) GET K8s pod readyReplicas → status updated to "Running" when pod is ready │ ▼ GET /sessions/{id} → {status: "Running", accessUrl: "https://…/s/<slug>/"}

The status lifecycle visible to all consumers (API, UI, background workers):

StatusMeaningSet by
StartingSession created in DB; Terraform apply in progressSessionLauncher (on enqueue)
RunningPod has ≥1 ready replica; accessUrl is populatedSessionStatusPoller
StoppingDELETE called; teardown queued to Service BusSessionEndpoints (DELETE handler)
StoppedK8s resources destroyed; session endedSessionStopConsumer
FailedTerraform apply threw an exceptionSessionLaunchWorker (catch block)

3.5 Per-User Storage Isolation

All researchers in a workspace share one Azure File Share (workspace-share) provisioned by TRE. Isolation is enforced at the Kubernetes volume mount level using the CSI driver's sub_path feature:

workspace-share (Azure File Share — TRE provisioned) ├── alice-research/ ← sub_path for user "alice-research" │ ├── my_notebook.ipynb │ └── results/ └── bob_dev/ ← sub_path for user "bob_dev" └── analysis.R Each session pod mounts only its own sub_path directory. Users cannot navigate above their mount point to reach another user's directory.

The sub_path value is a sanitised form of the user ID: all characters except [a-z0-9._-] are replaced with -. The subdirectory is created automatically on first mount — no pre-provisioning required.

3.6 React UI Session Flow

The React UI introduces a state machine for each application card. The UI makes no assumptions about provisioning time — it shows progressive feedback throughout.

UI StateVisualUser Action Available
IdleApp icon + name + Launch buttonClick Launch
LaunchingSpinner overlay, "Requesting…"None (button disabled)
StartingSpinner overlay, "Starting… (may take a few minutes)"None (button disabled)
RunningGreen active badge, Open button, Stop buttonOpen in new tab / Stop
StoppingSpinner overlay, "Stopping…"None
FailedRed error badge + error messageRetry

Idempotency: If a researcher clicks Launch while a session is already Starting or Running, the API returns the existing session (no duplicate deployment). The UI's click handler is disabled during Launching and Starting states as an additional safeguard.

4. Implementation Plan

The implementation is structured into five sequential phases. Each phase is independently deployable and testable. Phases 1–3 are backend; Phase 4 is UI; Phase 5 is cross-cutting E2E verification.

Phase 1  ·  AKS Namespace Provisioning  ·  ~14 hours

Goal: After Azure TRE provisions a workspace's Storage Account and File Share, automatically create the matching AKS namespace with network isolation and the storage credential Secret needed by the CSI driver.

Why first: All subsequent session launches depend on the namespace and CSI Secret existing. Nothing else can proceed without this.

  • T-01 — Terraform workspace-k8s module: kubernetes_namespace, azure-storage-creds Secret, default-deny NetworkPolicy
  • T-08 — TerraformRunner: backend.tfbackend file, init-skip optimisation, process kill on cancel
  • T-09 — TerraformOrchestrator.EnsureWorkspaceNamespaceAsync + DeleteWorkspaceNamespaceAsync

Exit criteria: After workspace activation, kubectl get ns shows the workspace namespace with azure-storage-creds Secret and default-deny policy in place.

Phase 2  ·  Application Catalogue & Domain Layer  ·  ~8 hours

Goal: Introduce the WorkspaceApplication and UserSession domain entities, the ISessionProvisioningQueue interface, the SessionLauncher service, and the ApplicationEndpoints API so workspace admins can register applications.

  • T-04 — WorkspaceApplication entity
  • T-05 — UserSession entity + SessionStatus enum
  • T-06 — ISessionProvisioningQueue interface (Core layer)
  • T-07 — SessionLauncher: validate → persist Starting session → enqueue → return immediately
  • T-15 — ApplicationEndpoints: POST / GET applications

Exit criteria: A workspace admin can register a Jupyter application via POST /api/workspaces/{id}/applications and retrieve it via GET.

Phase 3  ·  Session Launch Backend  ·  ~19 hours

Goal: Deliver the complete backend session lifecycle — launch (async), status polling, stop, and idle cleanup — backed by the session Terraform module.

  • T-02 — Terraform session module: Deployment + Service + Ingress + per-user sub_path
  • T-03 — Terraform app_configs.tf: Jupyter / RStudio / Custom startup logic and ingress rewrite
  • T-10 — TerraformOrchestrator.LaunchSessionAsync + StopSessionAsync + GetSessionStatusAsync
  • T-11 — SessionLaunchWorker: background channel worker, DI scope per item
  • T-12 — SessionStatusPoller: K8s readiness → Running transition
  • T-13 — SessionStopConsumer: Service Bus → terraform destroy → Stopped
  • T-14 — IdleSessionReaper: auto-stop beyond idle threshold
  • T-16 — SessionEndpoints: POST / GET / DELETE
  • T-17 — Program.cs: DI registration for all new services

Exit criteria: POST /sessions returns 202 in <2 s; GET /sessions/{id} transitions from Starting to Running once the pod is ready; DELETE tears down K8s resources cleanly.

Phase 4  ·  React UI  ·  ~21 hours

Goal: Integrate the session API into the existing React workspace application so researchers experience a seamless click-to-launch flow with live status feedback.

  • T-22 — sessionApi.ts: typed fetch wrappers (launchSession, getSession, listSessions, stopSession)
  • T-23 — useSessionPoller hook: polls GET at interval, stops on terminal state, cleans up on unmount
  • T-24 — AppCard component: 6 visual states (Idle / Launching / Starting / Running / Stopping / Failed)
  • T-25 — Workspace applications page: gallery of AppCards, load existing sessions on mount
  • T-26 — Session launch flow: click handler state machine — POST → Starting → Running/Failed
  • T-27 — Active sessions panel: list of Running sessions with Open and Stop actions
  • T-28 — Session stop flow: confirmation dialog → DELETE → card reset to Idle

Exit criteria: Researcher clicks a Jupyter icon, sees "Starting…" feedback, and the application opens in a new browser tab when ready. Running sessions persist across page reloads.

Phase 5  ·  End-to-End Testing  ·  ~12 hours

Goal: Verify all flows against a real AKS cluster and TRE-provisioned workspace before sign-off.

  • T-18 — E2E: Jupyter session launch, browser access, file persistence
  • T-19 — E2E: RStudio session, path prefix correctness, UI verification
  • T-20 — E2E: Per-user storage isolation (two researchers, same workspace)
  • T-21 — E2E: Session stop (DELETE) and idle reaper cleanup

Exit criteria: All four E2E scenarios pass against a live cluster. No orphaned K8s resources after stop. Storage isolation confirmed with two concurrent users.

6. Key Technical Decisions

6.1 Terraform as the Infrastructure Backend

Rather than imperative Azure SDK / Kubernetes client calls, all infrastructure operations are declared as Terraform modules (azurerm + kubernetes providers). The API shells out to the Terraform CLI.

BenefitDetail
Declarative drift detectionTerraform state tracks what was created; re-applying is idempotent
Destroy is symmetricterraform destroy removes exactly what apply created — no manual cleanup code
Remote stateAzure Blob backend; each workspace/session has its own .tfstate file — no state contention
Init-skip optimisationterraform init is skipped if .terraform/ already exists, saving 20–60 s per repeated operation

6.2 Async Session Launch (202 Accepted Pattern)

Terraform provisioning takes 2–3 minutes. Blocking the HTTP request would cause client timeouts and poor UX. The pattern used:

  • HTTP handler does only synchronous work: validate, persist Starting session, enqueue. Returns 202 Accepted in < 2 seconds.
  • SessionLaunchWorker (in-process Channel<T>) dequeues and runs Terraform in the background.
  • SessionStatusPoller polls K8s pod readiness and updates the session to Running.
  • Client polls GET /sessions/{id} — typically every 5 seconds — until Running or Failed.

6.3 In-Process Channel vs. Service Bus for Session Launch

Session launch uses an in-process System.Threading.Channels queue; session stop uses Service Bus. The distinction is intentional:

Session LaunchSession Stop
MechanismIn-process Channel<T>Azure Service Bus
ReasonInitiated from the same process; sub-millisecond enqueue; no durability needed — a restart with a Starting session is cleaned up by the poller or reaperMust survive API pod restart mid-teardown; Service Bus provides at-least-once delivery and dead-lettering

6.4 ISessionProvisioningQueue in Core, Not Infrastructure

SessionLauncher (Core layer) must not reference Infrastructure types. Defining ISessionProvisioningQueue in SRW.Core.Abstractions keeps the Clean Architecture dependency arrow pointing inward. SessionLaunchWorker (Infrastructure) implements it.

6.5 Ingress Path Routing: Jupyter vs. RStudio

The two supported app types require different nginx behaviour because they handle their own public path differently:

App TypeIngress RewriteWhy
JupyterNo rewrite — path passed through as-isJupyter accepts --NotebookApp.base_url=/s/<slug>/ at startup and generates all URLs with the prefix already included
RStudiorewrite-target: /$2 (strips prefix)RStudio does not support a --base-url flag; the prefix is injected via www-root-path in rserver.conf before startup; nginx must strip the prefix before forwarding
CustomNo rewrite (default)Supports __BASE_URL__ token substitution in commandJson

7. Risks & Mitigations

#RiskLikelihoodImpactMitigation
R-01 Terraform apply duration (~2-3 min) frustrates researchers expecting instant launch High Medium Async API (202 pattern) + UI progress indicator eliminates blocking wait. Terraform provider plugin cache (TF_PLUGIN_CACHE_DIR) and init-skip reduce cold-start time significantly.
R-02 Storage account key exposed in Terraform state or working directory Medium High Key passed exclusively as TF_VAR_storage_account_key environment variable — never written to .tfvars. Remote state in Azure Blob with RBAC-controlled access.
R-03 Session stuck in Starting if API pod restarts mid-provisioning Medium Medium IdleSessionReaper detects Starting sessions older than the configured threshold and marks them Failed, freeing the researcher to retry. Working directory on persistent storage allows Terraform resume.
R-04 Per-user storage isolation failure — users accessing each other's data Low High sub_path is enforced at the CSI driver / OS level, not application level. E2E test T-20 explicitly verifies isolation with two concurrent users before sign-off.
R-05 nginx-ingress annotation configuration-snippet blocked by cluster admission webhook Medium Low This implementation uses only standard annotations (rewrite-target, proxy-read-timeout) — no configuration-snippet. Known cluster restriction documented in platform runbook.
R-06 Terraform state file contention if two launches for the same session fire simultaneously Low Medium Idempotency check in SessionLauncher returns the existing session before enqueueing; single-reader channel ensures only one worker processes each session ID. Azure Blob backend provides state locking.
R-07 Data Protection key ring machine-local — cannot decrypt existing secrets on a new API pod Medium Medium Migrate Data Protection key ring to Azure Blob + Key Vault (shared store) before scaling API to multiple replicas. Documented as a follow-on hardening item.
